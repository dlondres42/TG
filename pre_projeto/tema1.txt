Methodological Rigor in Benchmarking Protocols
A graduation project focused on inference optimization must rely on a meticulously rigorous, reproducible, and statistically sound benchmarking methodology.45 Implementing naive timing scripts—such as measuring the difference utilizing time.time() before and after a Python .predict() call—is a fundamentally flawed approach. Such rudimentary methods are highly susceptible to operating system interruptions, background processes, and framework overhead, yielding invalid scientific data.47 Proper, defensible benchmarking requires a profound understanding of distinct performance metrics and the execution of stringent, isolated testing protocols.46
Defining the Axes of Evaluation: Latency and Throughput
System performance in machine learning delivery is universally evaluated along two inherently competing axes: latency and throughput.48
Throughput: Formally measured in requests per second (RPS) or inferences per second.3 It quantifies the absolute total volume of data the system can successfully process within a fixed, predefined time window.48 Throughput is systematically maximized by increasing batch sizes, effectively saturating the hardware resources and transforming a memory-bandwidth-bound operation into a highly efficient compute-bound operation.41
Latency: The true end-to-end duration from the exact microsecond a client dispatches a request to the moment the serialized response is fully received.39 It is imperative to understand that latency is fundamentally a statistical distribution, never a single, absolute value.50
In both academic research and industrial performance benchmarking, median latency (denoted as P50) is reported as the standard baseline expectation, representing the speed at which 50% of all requests are successfully processed.39 However, the most critical metric for evaluating system stability is tail latency, universally represented as the 95th percentile (P95) or the 99th percentile (P99).50 The P99 latency indicates the maximum time taken by the fastest 99% of requests; conversely, it highlights the severe delays experienced by the slowest 1%.50 Tail latency captures the rare, unpredictable system anomalies that cause severe processing delays, such as garbage collection pauses within the Python or Java runtime environment, network packet loss requiring TCP retries, unexpected context switching by the CPU scheduler, or sudden L3 cache misses.50 A truly robust and optimized ML delivery system is characterized not merely by a fast P50 median, but by a remarkably tight statistical grouping between the P50 and P99 metrics. This tight grouping represents low system jitter, high predictability, and operational excellence under load.50
Standardized Measurement Protocols
To guarantee absolute empirical validity and prevent configuration drift, an inference benchmark must surgically isolate the specific hardware and software variables being tested.52 A rigorous protocol involves distinct, sequential phases:
Hardware Initialization and Pinning: Ensuring that CPU affinity is strictly managed. For instance, binding specific inference threads to specific physical CPU cores utilizing commands like taskset in Linux.53 This prevents the operating system kernel from arbitrarily migrating the workload across different cores, which completely invalidates cache locality and introduces severe measurement noise.53
The Warm-up Phase: Before any performance measurements are recorded, the load testing client must send a barrage of dummy requests (e.g., 20 to 100 initial forward passes) to the endpoint.46 This critical phase allows the Just-In-Time (JIT) compilers to initialize, allocates the necessary memory buffers on the GPU, and populates the hardware L1/L2 caches.46 Measuring a "cold start" without explicitly intending to study cold starts will irreparably skew the P99 latency metrics, resulting in scientifically invalid conclusions.42
The Measurement Phase: The load generator dispatches simulated client requests concurrently for a sustained duration (e.g., 60 seconds of continuous load) or a fixed number of massive iterations.53 Industrial-grade tools such as Apache JMeter, wrk, or Locust must be utilized because they manage connection pooling efficiently and record highly accurate latency histograms at the microsecond level.37
Avoiding Coordinated Omission: A critical, often-overlooked pitfall in benchmarking is coordinated omission.47 This occurs when the load testing tool waits for a delayed response from the server before sending the next scheduled request.47 This inadvertently masks the severity of system stalls, as the tool backs off exactly when it should be measuring failure. Professional load generators must be configured to maintain a constant, unyielding request rate regardless of the server's response time to accurately capture queuing delays and expose true tail latency.47
Public Datasets for Reproducible Benchmarking
The fundamental reproducibility of a graduation project relies heavily on the utilization of open-access, highly standardized datasets.56 While image and text datasets dominate deep learning research (such as the ImageNet benchmark), tabular datasets present highly unique structural challenges for decision forests, characterized by wildly varying degrees of cardinality, sparsity, and dimensionality.7 To comprehensively benchmark tree ensembles and eliminate bias, the following public datasets are considered the absolute industry standards within the DFInferBench framework and similar academic testing suites 8:
The Higgs Dataset (UCI Machine Learning Repository): A massively dense dataset containing 11 million instances and 28 numerical features, originally utilized for binary classification in high-energy physics.57 Because it contains absolutely no missing values and a relatively small feature space, it is the premier benchmark for testing raw, compute-bound execution speed on CPUs and GPUs.58
The Airline Dataset: Contains over 115 million comprehensive records of commercial flight data utilized to predict arrival delays.11 It introduces profound complexity through a dense mixture of categorical and continuous variables, serving as a robust, real-world test for data preprocessing algorithms and categorical encoding efficiency prior to inference.11
The Epsilon Dataset: A highly dense, extremely wide synthetic dataset where each individual instance possesses 2,000 features.8 It is explicitly utilized to test the memory bandwidth limitations and cache capacity of the inference engine, as moving large feature vectors across the system bus becomes the primary processing bottleneck, completely overshadowing the mathematical computation.8
The Criteo Terabyte Dataset: A massive, terabyte-scale dataset utilized extensively in the advertising industry for click-through rate (CTR) prediction.57 It contains approximately 1 million features per tuple and is highly sparse, riddled with missing values.8 Benchmarking on Criteo evaluates how efficiently an inference engine and delivery framework handles complex memory pointers and sparse matrix representations.8

Proposed Graduation Project Themes
Drawing upon the sophisticated complexity established by recent CIn-UFPE graduation works 6, the following five project themes meticulously integrate inference optimization frameworks, modern serving architectures, and rigorous benchmarking protocols. Each theme is purposely designed to have a short, clearly defined academic scope while yielding high-value, empirically sound data that directly addresses current MLOps challenges.
Theme 2: Tail Latency (P99) Optimization in Serverless ML Delivery for Tabular Data
Context: Industrial microservice architectures are rapidly migrating toward serverless or auto-scaling container deployments orchestrated by Kubernetes.37 In these highly dynamic environments, "cold starts" and system jitter severely impact the end-user experience, rendering median latency (P50) an entirely insufficient metric for system reliability.42 Objective: To isolate, measure, and minimize the specific software causes of P99 tail latency when serving optimized tabular machine learning models through lightweight Python web frameworks versus specialized ML servers. Methodology: An XGBoost model is trained on the Airline dataset to handle mixed data types.57 The model is explicitly exported to the standardized ONNX format to guarantee absolute interoperability between servers.10 Two distinct delivery pipelines are constructed: one utilizing a standard FastAPI asynchronous application, and one utilizing BentoML.40 A rigorous benchmarking protocol is executed, specifically designed to measure cold start initialization times and P99 latency distributions.42 The benchmark manipulates the batch size parameters within BentoML’s adaptive batching engine to observe the effect on the latency tail. Expected Deliverable: A system architecture blueprint and quantitative statistical analysis proving that while FastAPI may provide low P50 latency for isolated individual requests, adaptive batching within specialized runtimes forcibly compresses the P99 latency tail, offering vastly superior statistical reliability under highly variable traffic conditions.41

The optimization of machine learning inference represents a vital intersection, effectively bridging the theoretical algorithms of data science with the highly demanding physical realities of robust software and systems engineering. As clearly demonstrated by the escalating technical complexity of recent CIn-UFPE graduation works, the academic evaluation of modern ML systems demands a rigorous, empirical understanding of systemic behavior under operational load.6 By purposefully shifting the research focus away from the highly saturated domain of model training and strictly analyzing the mechanics of model delivery, an undergraduate thesis can yield profound, highly actionable insights into the mechanics of modern computing.
Focusing the scope on tree ensembles provides a mathematically stable, highly interpretable, and reproducible baseline.4 Through the integration of standardized public tabular datasets 57 and state-of-the-art computational frameworks—such as the tensor-based graph execution of Hummingbird, the ahead-of-time C-compilation of Treelite, and the GPU predicate optimization of Nvidia FIL—a researcher can construct an extensive and highly valid empirical study.8 By further combining these deep algorithmic optimizations with industrial serving platforms like BentoML or the Triton Inference Server 41, and measuring true performance utilizing strict P99 latency percentiles to account for system jitter 50, the resulting graduation project will not only fulfill stringent academic requirements but will directly and competently address the most pressing scalability challenges present in the contemporary software industry.
